1.1 创建版本库并提交

开始使用git之前，需要设置一下Git的配置变量（设置1次即可）

1.设置姓名和邮箱(必须)

git config --global user.name = "liugenpeng"

git config --global user.email = "liugenpeng@beisen.com"

2.设置一些命令别名（可选）

git config --system alias.st status

git config --system alias.ci commit

3.开启颜色显示（可选）

git config --global color.ui true

创建版本库

1） 第一种方式： cd 某一目录 , 然后执行 git init
2） 第二种方式： git init 目录名（针对高版本）

为什么工作区根目录下有一个.git目录

这是git与其他分布式版本控制系统的一个显著特点。对于git而言，版本库位于根目录下的.git目录中，仅此一处。

传统集中式版本控制系统的版本库和工作区是分开的，甚至是在不同主机，因此必须建立工作区和版本库的对应。

下面是对比：

CVS:工作区的根目录及每一个子目录都有一个CVS目录，CVS目录包括几个配置文件，通过这些配置文件建立对版本库的追踪。例如CVS目录下的Entries文件记录了从版本库检出到工作区文件的名称、版本和时间戳等，通过时间戳的对比可以快速扫描工作区内文件的改动。优点：工作区可以移动到任何其他目录；缺点是工作区文件修改了，只能对整个文件进行传输，因为没有原始文件做对比，导致传输效率降低，还有会造成信息泄露（Web服务器目录下如果包含CVS目录，黑客可以扫描CVS\Entries文件得到目录的文件列表）。

SVN:工作区的根目录和每一个子目录下都有一个.svn目录。目录.svn包含了类似于CVS的配置文件和当前工作区下每一个文件的拷贝。优点是可以让某些svn子命令脱离版本库执行，文件修改后只提交修改的部分。缺点是信息泄露，以及加倍占用工作空间。
。
git:使所有版本控制操作都可以在本地即可完成，只要保护好.git目录，不会造成安全泄漏；git 提供了一条geit grep命令搜索工作区的文件内容.

$ git grep "hello"

welcome.txt:hello

这样说一下，Git的相关操作不一定要在工作区的根目录下执行，因为在git工作区的某个子目录下执行操作的时候，会在工作区目录中依次想上递归查找.git目录，.git目录下的index记录了工作区文件的状态（暂存区的状态）。
